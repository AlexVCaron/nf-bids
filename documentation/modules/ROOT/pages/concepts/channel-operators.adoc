= Closure-Based Channel Operators
:navtitle: Closure-Based Channel Operators

[.lead]
nf-bids extends Nextflow with three closure-driven operators — `groupTupleBy`,
`joinBy`, and `combineBy` — that work with maps and arbitrary Groovy objects instead of
requiring fixed-position tuples.

== Why closure-based operators?

Nextflow's built-in `groupTuple`, `join`, and `combine` operators require _tuple index_
keys:

[source,groovy]
----
// Built-in: tuple structure required, index is opaque
channel.of(['sub-01', 'file.nii'], ['sub-01', 'other.nii'])
       .groupTuple(by: 0)
----

With nf-bids flat output, items are maps — not tuples.  Closure-based operators read
any field without restructuring:

[source,groovy]
----
// Closure-based: works with maps, self-documenting
channel.of([subject: 'sub-01', file: 'file.nii'],
           [subject: 'sub-01', file: 'other.nii'])
       .groupTupleBy { it.subject }
----

=== Importing the operators

[source,groovy]
----
include { groupTupleBy } from 'plugin/nf-bids'
include { joinBy }       from 'plugin/nf-bids'
include { combineBy }    from 'plugin/nf-bids'
----

== groupTupleBy

=== Signature

[source,groovy]
----
channel.groupTupleBy(Closure keyExtractor)
channel.groupTupleBy(Closure keyExtractor, Map opts)
----

=== Output

Emits `[key, [items]]` tuples.

=== Options

[cols="1,1,3"]
|===
| Option | Type | Description

| `size`
| `Integer`
| Emit a group as soon as it reaches this many items (streaming groups).

| `sort`
| `Boolean` or `Closure`
| `true` for natural order; a two-argument closure for custom comparator.

| `remainder`
| `Boolean`
| When `true` (default), emit incomplete groups when the channel closes.
|===

=== Examples

.Group by single field
[source,groovy]
----
Channel
    .of([subject: 'sub-01', file: 'a.nii'],
        [subject: 'sub-01', file: 'b.nii'],
        [subject: 'sub-02', file: 'c.nii'])
    .groupTupleBy { it.subject }
// → ['sub-01', [[subject:'sub-01', file:'a.nii'],
//               [subject:'sub-01', file:'b.nii']]]
//   ['sub-02', [[subject:'sub-02', file:'c.nii']]]
----

.Composite key (subject + session)
[source,groovy]
----
channel.groupTupleBy { [it.subject, it.session] }
// Closure returns a List → wrapped automatically in CompositeKey
// → [[sub-01, ses-01], [ [item1, item2] ]]
----

.Sort items within each group
[source,groovy]
----
channel.groupTupleBy({ it.subject }, [sort: { a, b -> a.run <=> b.run }])
----

.Streaming groups of 3
[source,groovy]
----
channel.groupTupleBy({ it.batch }, [size: 3, remainder: false])
----

== joinBy

=== Signature

[source,groovy]
----
left.joinBy(DataflowReadChannel right, Closure keyExtractor)
left.joinBy(DataflowReadChannel right, Closure leftKey, Closure rightKey)
left.joinBy(DataflowReadChannel right, Closure leftKey, Closure rightKey, Map opts)
left.joinBy(DataflowReadChannel right, Closure leftKey, Map opts)
----

When a single `keyExtractor` is supplied it is applied to items from both channels.

=== Output

Emits `[key, leftItem, rightItem]` tuples for every matching pair.

=== Options

[cols="1,1,3"]
|===
| Option | Type | Description

| `remainder`
| `Boolean`
| When `true`, emit unmatched items with `null` as the missing partner
  (outer-join semantics).

| `failOnDuplicate`
| `Boolean`
| Fail if the same key appears more than once in either channel.

| `failOnMismatch`
| `Boolean`
| Fail if a key appears in one channel but not the other.
|===

=== Examples

.Join anatomical and functional by subject
[source,groovy]
----
anatomical = Channel.of([subject: 'sub-01', file: 't1.nii'],
                        [subject: 'sub-02', file: 't1.nii'])
functional = Channel.of([subject: 'sub-01', file: 'bold.nii'],
                        [subject: 'sub-02', file: 'bold.nii'])

anatomical.joinBy(functional) { it.subject }
// → ['sub-01', [subject:'sub-01', file:'t1.nii'],
//              [subject:'sub-01', file:'bold.nii']]
//   ['sub-02', ...]
----

.Different key fields on each channel
[source,groovy]
----
scans.joinBy(metadata,
             { it.scan_id },        // left key
             { it.participant_id }) // right key
----

.Outer join (include unmatched)
[source,groovy]
----
left.joinBy(right, { it.id }, [remainder: true])
// Unmatched items emitted as [key, item, null] or [key, null, item]
----

== combineBy

=== Signature

[source,groovy]
----
left.combineBy(DataflowReadChannel right, Closure keyExtractor)
left.combineBy(DataflowReadChannel right, Closure keyExtractor, Map opts)
left.combineBy(DataflowReadChannel right, Closure leftKey, Closure rightKey, Map opts)
----

=== Output

Emits `[key, leftItem, rightItem]` tuples — the full _cartesian product_ of matching
items within each key group (inner join with product).

=== Examples

.Match subjects with sessions
[source,groovy]
----
subjects = Channel.of([id: 'sub-01', age: 25], [id: 'sub-02', age: 30])
sessions = Channel.of([id: 'sub-01', session: 'ses-01'],
                      [id: 'sub-01', session: 'ses-02'],
                      [id: 'sub-02', session: 'ses-01'])

subjects.combineBy(sessions, { it.id })
// → ['sub-01', [id:'sub-01', age:25], [id:'sub-01', session:'ses-01']]
//   ['sub-01', [id:'sub-01', age:25], [id:'sub-01', session:'ses-02']]
//   ['sub-02', [id:'sub-02', age:30], [id:'sub-02', session:'ses-01']]
----

.Filter after combining
[source,groovy]
----
subjects
    .combineBy(sessions, { it.id })
    .filter { key, subj, sess -> subj.age >= 18 }
----

== CompositeKey and KeyExtractor

When a closure returns a `List`, `KeyExtractor.extractKey()` wraps it in
`CompositeKey` — an `@EqualsAndHashCode`-annotated class — before it is used as a
`HashMap` key.  This ensures `[sub-01, ses-01] == [sub-01, ses-01]` even across
different `List` instances.

[source,groovy]
----
channel.groupTupleBy { [it.subject, it.session] }
// Returns List → becomes CompositeKey([sub-01, ses-01]) internally
----

`KeyExtractor.validateKeyExtractor(closure, operatorName)` checks that the closure
accepts at least one parameter.  Zero-parameter closures raise
`IllegalArgumentException` at operator construction time.

== Performance notes

* `groupTupleBy` buffers all items in memory until the channel closes (or `size` is
  reached); for large datasets use `size` to stream groups.
* `joinBy` performs cartesian product for duplicate keys; 1:N joins are fine, M:N
  should be avoided in hot paths.
* `combineBy` always produces a full cartesian product; guard with `.filter()` to
  reduce downstream load.
