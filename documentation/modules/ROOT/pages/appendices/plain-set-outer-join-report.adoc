= Plain-Set Outer-Join Behaviour Report
:navtitle: Plain-Set Outer-Join Report

[.lead]
This report documents an incorrect "fix" that was introduced to
`PlainSetHandler`, why it was wrong, what it did to the data pipeline,
and the rationale for reverting it.
The revert was made in commit `revert: remove run-entity guard from PlainSetHandler; add dir-variant test`.

== Background: What `plain_set` Does

A `plain_set` entry in `bids2nf.yaml` tells the plugin:
"for each loop-over group, collect every primary file whose suffix matches this key and emit one Nextflow channel item per file."

That definition is both simple and general.
`PlainSetHandler.processGroup` iterates over the collected `primaryFiles` list and emits one `BidsChannelData` per entry.
Each emitted item carries:

* the primary file path (e.g. `fmap/sub-01_ses-01_dir-AP_epi.nii.gz`),
* its matched sidecars (JSON, bval/bvec, …) keyed by extension type, and
* all BIDS entities extracted from the filename.

The design is deliberately entity-agnostic: the handler does not know or care which entity distinguishes the files within a group; it just emits them all.

== The Broken Fix (Reverted)

=== What It Added

[source,groovy]
----
Set<String> runValues = primaryFiles.collect { f ->
    f.getEntityValue(BidsEntity.normalizeName('run')) ?: ''
}.toSet()

if (runValues.size() <= 1) {
    primaryFiles = [primaryFiles.first()]
}
----

=== What It Said It Did

The comment claimed that an outer-join (one item per primary file) is "only meaningful when files within the same loop-over group differ by the `run` entity."
For any other variation it kept only the lexicographically smallest path.

=== Data Summary

[cols="1,2,2"]
|===
| Scenario | Before fix | After fix (broken state)

| `sub-01_ses-005_dir-AP_epi.nii.gz` + `sub-01_ses-005_dir-PA_epi.nii.gz`
  (no `run` entity)
| 2 channel items emitted — one per direction
| 1 channel item emitted — `dir-PA` silently dropped

| `sub-01_ses-01_run-01_epi.nii.gz` + `sub-01_ses-01_run-02_epi.nii.gz`
  (two distinct `run` values)
| 2 items
| 2 items (guard did not trigger)

| Single `sub-01_T1w.nii.gz` (no run, no dir)
| 1 item
| 1 item (guard did not trigger — `runValues = {''}`, size == 1, first() == only file)

|===

In the `dir-AP / dir-PA` case the guard caused **silent data loss**: the downstream pipeline received half the EPI volumes without any warning or error.

== Root-Cause Analysis

=== False premise: `run` is a general discriminator

The BIDS specification does not assign a special grouping role to the `run` entity.
Any entity that is absent from `loop_over` can serve as the axis of variation within a loop-over group.
Common cases include:

[cols="1,2"]
|===
| Entity | Typical use

| `run` | Repeated acquisitions (runs 1, 2, 3…)
| `dir` | Phase-encoding direction (AP/PA, LR/RL)
| `part` | Image part (magnitude/phase)
| `echo` | Echo index in multi-echo sequences
| `acq` | Acquisition protocol variant
| `ce` | Contrast-agent variant

|===

`PlainSetHandler` is a suffix matcher, not a run matcher.
Filtering on `run` uniqueness breaks every case where a different entity is the discriminating axis.

=== Why the fix appeared to work

The `ds-dwi3` test dataset contains two EPIs that differ by `dir` but have no `run` entity.
When the guard was introduced it collapsed them to one item, which matched the (incorrect) test expectation.
Separately, the JSON sidecar files for that dataset were missing from the repository, causing unrelated failures.
The two bugs masked each other: restoring the JSON files + applying the guard made the overall test suite green, but the guard itself was encoding wrong behaviour.

=== Correct resolution

. Restore the missing JSON sidecar files in `test/data/custom/ds-dwi3/` (kept).
. Remove the `run` guard entirely so that `processGroup` emits one item per collected primary file unconditionally (reverted).
. Add a new unit test asserting that two EPIs differing only by `dir` each produce their own channel item.

== State After the Revert

`PlainSetHandler.processGroup` now:

. Sorts all collected `primaryFiles` by normalized path (deterministic ordering).
. Iterates and emits one `BidsChannelData` per file — no filtering, no entity-specific logic.
. Attaches only the sidecars whose `getBasename()` matches the current primary file, preventing cross-contamination between, e.g., `dir-AP.json` and `dir-PA.json`.

=== Guarantees preserved

[cols="1,2"]
|===
| Property | Status

| Determinism | ✓ — files sorted by normalized path before emission
| Sidecar isolation | ✓ — basename-matching prevents cross-file sidecar bleed
| No silent data loss | ✓ — all matched primary files are emitted
| Entity-agnostic | ✓ — no special casing of any BIDS entity

|===

== Downstream Implications

Pipelines that call `Channel.fromBIDS` with a `plain_set` suffix may now receive more channel items than before this revert if their dataset contains files that differ by a non-`run` entity within the same loop-over group.
This is the correct and intended behaviour.
Any downstream code that assumed at most one item per suffix (e.g. by calling `.first()` on the resulting channel without a prior grouping step) should be reviewed.
